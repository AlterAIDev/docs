## Overview

SCIM (System for Cross-domain Identity Management) 2.0 enables your identity provider to automatically push user and group changes to Alter Vault. When a user is deprovisioned in your IDP, their OAuth grants are **immediately revoked** — no polling or delay.

<Info>
**Security critical**: SCIM deprovisioning is the fastest way to revoke access when an employee leaves your organization. JWT-based sync only triggers on the next token request, while SCIM pushes the change in real time.
</Info>

## Enabling SCIM

<Steps>

### Navigate to Identity Provider Settings

In the **Developer Portal**, go to your application's **Identity** page, then select the **SCIM** tab.

### Enable SCIM

Click **Enable SCIM** to generate a bearer token and SCIM endpoint URL.

### Copy Your Credentials

You'll receive:
- **SCIM Endpoint URL** — The base URL your IDP will push to
- **Bearer Token** — The authentication token for SCIM requests

<Warning>
**The bearer token is shown only once!** Copy it immediately and store it securely. If you lose it, you can rotate it from the Developer Portal (which invalidates the previous token).
</Warning>

### Configure Your IDP

Configure your identity provider to push SCIM events to the endpoint URL using the bearer token. See [IDP Setup Guides](#idp-setup-guides) below for provider-specific instructions.

</Steps>

## Authentication

All SCIM requests must include a bearer token in the `Authorization` header:

```
Authorization: Bearer <your_scim_token>
```

## Supported Operations

| Operation | Method | Path | Description |
|---|---|---|---|
| **List Users** | GET | `/Users` | Filter and paginate users |
| **Get User** | GET | `/Users/{id}` | Get a single user |
| **Create User** | POST | `/Users` | Provision a new user |
| **Update User** | PATCH | `/Users/{id}` | Update user attributes |
| **Replace User** | PUT | `/Users/{id}` | Full user replacement |
| **Delete User** | DELETE | `/Users/{id}` | Remove user and revoke access |
| **List Groups** | GET | `/Groups` | List provisioned groups |
| **Get Group** | GET | `/Groups/{id}` | Get a single group |
| **Create Group** | POST | `/Groups` | Create a group |
| **Update Group** | PATCH | `/Groups/{id}` | Add/remove group members |
| **Delete Group** | DELETE | `/Groups/{id}` | Delete group |

### Discovery Endpoints

| Endpoint | Description |
|---|---|
| `GET /ServiceProviderConfig` | SCIM capabilities and supported features |
| `GET /Schemas` | Schema definitions for User and Group resources |
| `GET /ResourceTypes` | Resource type metadata |

## IDP Setup Guides

<Tabs>

<Tab title="Okta">

### Okta SCIM Provisioning

1. In Okta Admin Console, go to **Applications** → your application
2. Go to **Provisioning** tab → **Configure API Integration**
3. Check **Enable API Integration**
4. Enter:
   - **SCIM connector base URL**: Your SCIM endpoint URL from the Developer Portal
   - **Unique identifier field for users**: `userName`
   - **Authentication Mode**: HTTP Header
   - **Authorization**: Paste your bearer token
5. Click **Test API Credentials** to verify the grant
6. Under **Provisioning** → **To App**:
   - Enable **Create Users**
   - Enable **Update User Attributes**
   - Enable **Deactivate Users**
7. Click **Save**

<Info>
Okta sends `PATCH` with `active: false` when a user is unassigned or deactivated, which triggers the deprovisioning cascade in Alter Vault.
</Info>

</Tab>

<Tab title="Azure AD (Entra ID)">

### Microsoft Entra ID SCIM Provisioning

1. In Azure Portal, go to **Enterprise Applications** → your application
2. Go to **Provisioning** → **Get started**
3. Set **Provisioning Mode** to **Automatic**
4. Under **Admin Credentials**:
   - **Tenant URL**: Your SCIM endpoint URL from the Developer Portal
   - **Secret Token**: Paste your bearer token
5. Click **Test Connection** to verify
6. Under **Mappings**, configure:
   - **Provision Azure Active Directory Users** → Enabled
   - **Provision Azure Active Directory Groups** → Enabled
7. Review attribute mappings (defaults work for most setups)
8. Set **Provisioning Status** to **On**
9. Click **Save**

<Info>
Azure AD runs provisioning cycles every 40 minutes by default. Initial provisioning may take longer. You can trigger an on-demand sync from the Provisioning page.
</Info>

</Tab>

<Tab title="Keycloak">

### Keycloak SCIM Plugin

Keycloak requires a SCIM plugin (not built-in):

1. Install the [keycloak-scim](https://github.com/Captain-P-Goldfish/scim-for-keycloak) plugin
2. In Keycloak Admin Console, go to **Realm Settings** → **SCIM** (from plugin)
3. Add a new SCIM service provider:
   - **URL**: Your SCIM endpoint URL
   - **Authentication**: Bearer Token
   - **Token**: Paste your bearer token
4. Configure user and group provisioning
5. Enable the integration

</Tab>

</Tabs>

## Deprovisioning

When a user is deprovisioned via SCIM (either `PATCH active=false` or `DELETE`), the following happens immediately:

<Warning>
**Irreversible**: Deprovisioning permanently revokes all connections. If the user is re-provisioned later, they will need to re-authorize their OAuth grants.
</Warning>

1. **User status** is set to `deprovisioned`
2. **All OAuth grants** linked to the user are immediately revoked
3. **All stored tokens** are permanently deleted
4. **Subsequent token requests** for this user are denied

## Token Rotation

If you need to rotate the SCIM bearer token (e.g., suspected compromise):

1. In **Developer Portal** → **Identity** → **SCIM** tab
2. Click **Rotate Token**
3. Copy the new bearer token (shown only once)
4. Update the token in your IDP's SCIM configuration
5. The previous token is immediately invalidated

<Info>
**Minimize downtime**: Update your IDP configuration with the new token immediately after rotation. SCIM requests with the old token will fail with `401 Unauthorized`.
</Info>

## Sync Status

Monitor SCIM sync status in the **Developer Portal** under your IDP's **SCIM** tab:

- **Last synced at** — When the last successful SCIM operation occurred
- **Users synced** — Total number of users provisioned via SCIM
- **Groups synced** — Total number of groups provisioned via SCIM
- **Last error** — Most recent SCIM error (cleared on successful sync)

## Troubleshooting

<AccordionGroup>

<Accordion title="SCIM requests return 401 Unauthorized">
- Verify the bearer token is correct and hasn't been rotated
- Ensure the `Authorization` header uses the `Bearer` prefix
- Check that SCIM is still enabled for the IDP in the Developer Portal
</Accordion>

<Accordion title="Users not syncing">
- Verify SCIM is enabled in the Developer Portal
- Check your IDP's SCIM provisioning configuration
- For Okta: ensure "Create Users" and "Update User Attributes" are enabled under Provisioning → To App
- For Azure AD: verify the Provisioning Status is "On"
</Accordion>

<Accordion title="Deprovisioning not working">
- Ensure your IDP sends `PATCH` with `active: false` (Okta/Azure AD default behavior)
- Check the SCIM sync status in Developer Portal for errors
- Verify the user has OAuth grants linked to them in Alter Vault
</Accordion>

<Accordion title="Groups not appearing">
- For Okta: enable "Push Groups" in the application provisioning settings
- For Azure AD: ensure group provisioning mapping is enabled
- Verify groups are assigned to the application in your IDP
</Accordion>

</AccordionGroup>
