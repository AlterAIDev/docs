## Prerequisites

Before you begin, make sure you have:

- An Alter Vault account with an application created in the Developer Portal
- An OIDC-compatible identity provider (Okta, Azure AD, Auth0, Clerk, Keycloak, etc.)
- The Python SDK or TypeScript SDK installed

## Setup Steps

<Steps>

### Add Your Identity Provider

In the **Developer Portal**, navigate to your application's **Identity** page.

1. Click **Add Identity Provider**
2. Enter your OIDC issuer URL (e.g., `https://your-org.okta.com`)
3. Alter Vault auto-detects the IDP type, user ID claim, and group claim
4. Optionally override the audience or claim mappings
5. Click **Save**

<Info>
**Auto-detection**: Alter Vault recognizes Okta, Azure AD, Auth0, Clerk, Keycloak, Cognito, Firebase, and Supabase from the issuer URL and automatically configures the correct claim mappings.
</Info>

### Pass JWT When Retrieving Tokens

When your application makes API calls through Alter Vault, include the user's JWT from your IDP. This enables Alter Vault to identify the user and enforce access policies.

<CodeGroup>

```python Python SDK
from alter_sdk import AlterVault, ActorType, HttpMethod

vault = AlterVault(
    api_key="alter_key_...",
    actor_type=ActorType.AI_AGENT,
    actor_identifier="my-agent",
    user_token_getter=lambda: get_user_jwt(),  # Returns JWT from your IDP
)

# JWT is automatically included by the SDK via user_token_getter
response = await vault.request(
    None,  # connectionId not needed with identity resolution
    HttpMethod.GET,
    "https://api.example.com/data",
    provider="google",
)
```

```typescript TypeScript SDK
import { AlterVault, ActorType, HttpMethod } from "@alter-ai/alter-sdk";

const vault = new AlterVault({
  apiKey: "alter_key_...",
  actorType: ActorType.AI_AGENT,
  actorIdentifier: "my-agent",
  userTokenGetter: () => getUserJwt(), // Returns JWT from your IDP
});

// JWT is automatically included by the SDK via userTokenGetter
const response = await vault.request(
  null, // connectionId not needed with identity resolution
  HttpMethod.GET,
  "https://api.example.com/data",
  { provider: "google" },
);
```

</CodeGroup>

<Info>
**Automatic sync**: The first time Alter Vault sees a JWT for a new user, it automatically creates a user record and syncs group memberships from the JWT claims. Subsequent requests update the user's profile.
</Info>

### Enable SCIM (Recommended for Enterprise)

For enterprise IDPs like Okta and Azure AD, enable SCIM for real-time user provisioning and immediate deprovisioning.

1. In **Developer Portal** → **Identity** → **SCIM** tab
2. Click **Enable SCIM**
3. **Copy the bearer token** (shown only once!) and the SCIM endpoint URL
4. Configure your IDP to push to this endpoint (see [SCIM Reference](/identity-providers/scim) for per-IDP instructions)

<Warning>
**Save the bearer token immediately!** The SCIM bearer token is displayed only once when you enable SCIM. If you lose it, you can rotate it from the Developer Portal, but the old token will be invalidated.
</Warning>

### Test Deprovisioning

To verify the integration works end-to-end:

1. Assign a test user to your application in your IDP
2. Verify the user appears in Alter Vault (check the SCIM sync status in Developer Portal)
3. Remove the test user from your IDP
4. Verify their OAuth connections are revoked in Alter Vault

</Steps>

## What Happens Behind the Scenes

When you pass a JWT or when your IDP sends a SCIM event:

1. **User identity is resolved** — Alter Vault maps the JWT `sub` claim (or SCIM `externalId`) to an internal user record
2. **Groups are synced** — Group memberships from JWT claims or SCIM group operations are synced to Alter Vault
3. **Connections are linked** — OAuth connections are associated with the user for lifecycle management
4. **Policies are evaluated** — Group-based access policies can restrict which providers a user can access

When a user is **deprovisioned**:

1. User status is set to `deprovisioned`
2. All linked OAuth connections are immediately revoked
3. All stored tokens are permanently deleted
4. Subsequent token requests for this user are denied
