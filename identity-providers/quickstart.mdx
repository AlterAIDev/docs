## Prerequisites

Before you begin, make sure you have:

- An Alter Vault account with an application created in the Developer Portal
- An OIDC-compatible identity provider (Okta, Azure AD, Auth0, Clerk, Keycloak, etc.)
- The Python SDK or TypeScript SDK installed

## Setup Steps

<Steps>

### Add Your Identity Provider

In the **Developer Portal**, navigate to your application's **Identity** page.

1. Click **Add Identity Provider**
2. Enter your OIDC issuer URL (e.g., `https://your-org.okta.com`)
3. Click **Discover** to auto-detect the IDP type, user ID claim, and group claim
4. Review the suggested claim mappings and override if needed
5. Click **Add Provider**

<Info>
**Auto-detection**: Alter Vault recognizes Okta, Azure AD, Auth0, Clerk, Google, Keycloak, Cognito, Firebase, and Supabase from the issuer URL and automatically configures the correct claim mappings.
</Info>

<Warning>
**Claim mappings lock after first user sign-in.** The user ID claim, group claim, and role claim cannot be changed once the first user authenticates through the identity provider. If the IDP does not include group or role claims by default (e.g., Auth0 requires a custom Action), leave these fields empty and set them before any user signs in.
</Warning>

### Pass JWT When Retrieving Tokens

When your application makes API calls through Alter Vault, include the user's JWT from your IDP. This enables Alter Vault to identify the user and enforce access policies.

<CodeGroup>

```python Python SDK
from alter_sdk import AlterVault, HttpMethod

vault = AlterVault(
    api_key="alter_key_...",
    caller="my-agent",
    user_token_getter=lambda: get_user_jwt(),  # Returns JWT from your IDP
)

# JWT is automatically included by the SDK via user_token_getter
response = await vault.request(
    HttpMethod.GET,
    "https://api.example.com/data",
    provider="google",
)
```

```typescript TypeScript SDK
import { AlterVault, HttpMethod } from "@alter-ai/alter-sdk";

const vault = new AlterVault({
  apiKey: "alter_key_...",
  caller: "my-agent",
  userTokenGetter: () => getUserJwt(), // Returns JWT from your IDP
});

// JWT is automatically included by the SDK via userTokenGetter
const response = await vault.request(
  HttpMethod.GET,
  "https://api.example.com/data",
  { provider: "google" },
);
```

</CodeGroup>

<Info>
**Automatic sync**: The first time Alter Vault sees a JWT for a new user, it automatically creates a user record and syncs group memberships from the JWT claims. Subsequent requests update the user's profile.
</Info>

### Enable User Authentication (OIDC)

To let end users sign in through the IDP --- for `vault.authenticate()` in the SDK or the Wallet dashboard --- configure OIDC client credentials.

1. In the **Developer Portal** → the app's **Identity** page, expand the **User Authentication (OIDC)** section
2. **Step 1 --- Register redirect URIs**: Copy both redirect URIs (SDK callback and Wallet callback) and add them as allowed redirect/callback URIs in the OAuth application on the IDP
3. **Step 2 --- Enter credentials**: Copy the **Client ID** and **Client Secret** from the IDP's OAuth application and paste them into the Developer Portal, then click **Save Credentials**

<Info>
**What this enables**: Once configured, the SDK's `vault.authenticate()` method opens a browser-based login flow through the IDP, and the Wallet dashboard gains a sign-in page for end users. User accounts are created automatically on first authentication.
</Info>

#### Per-Provider Setup

<Tabs>
<Tab title="Auth0">
1. Go to **Auth0 Dashboard → Applications → Applications** and click **Create Application**
2. Choose **Regular Web Application** (not SPA --- the callback goes to the Alter Vault backend)
3. In the app's **Settings** tab, paste both redirect URIs into **Allowed Callback URLs** (comma-separated)
4. Copy the **Client ID** and **Client Secret** from the same Settings page into the Developer Portal

<Warning>
If using a custom Auth0 domain (e.g., `auth.yourcompany.com`), the issuer URL registered in Alter Vault must match the issuer in the JWTs Auth0 issues. Check **Auth0 Dashboard → Settings → Custom Domains**.
</Warning>
</Tab>

<Tab title="Okta">
1. Go to **Okta Admin → Applications → Applications** and click **Create App Integration**
2. Choose **OIDC - OpenID Connect**, then **Web Application**
3. Under **Sign-in redirect URIs**, add both redirect URIs from the Developer Portal
4. Copy the **Client ID** and **Client Secret** from the app's **General** tab into the Developer Portal
</Tab>

<Tab title="Entra ID">
1. Go to **Azure Portal → App registrations** and select the application (or create a new one)
2. Go to **Authentication → Add a platform → Web**
3. Add both redirect URIs from the Developer Portal
4. Go to **Certificates & secrets → New client secret** and create a secret
5. Copy the **Application (client) ID** from the Overview page and the **client secret value** into the Developer Portal

<Warning>
When SCIM provisioning is enabled for Entra ID, the user ID claim must be set to `oid` (not `sub`). Entra ID SCIM sends the `objectId` as the user identifier, and a mismatch creates duplicate user records.
</Warning>
</Tab>

<Tab title="Google">
1. Go to **Google Cloud Console → APIs & Services → Credentials**
2. Click **Create Credentials → OAuth client ID**, select **Web application**
3. Under **Authorized redirect URIs**, add both redirect URIs from the Developer Portal
4. Copy the **Client ID** and **Client secret** into the Developer Portal
</Tab>

<Tab title="Keycloak">
1. Go to **Keycloak Admin → Clients** and click **Create client**
2. Set **Client type** to **OpenID Connect** and **Client authentication** to **On**
3. Under **Valid redirect URIs**, add both redirect URIs from the Developer Portal
4. Go to the **Credentials** tab and copy the **Client ID** and **Client secret** into the Developer Portal
</Tab>

<Tab title="Other OIDC">
1. Create an OAuth/OIDC application in the identity provider
2. Set the application type to **Web** or **Confidential** (not SPA/public)
3. Add both redirect URIs from the Developer Portal as allowed callback URLs
4. Copy the **Client ID** and **Client Secret** into the Developer Portal
</Tab>
</Tabs>

### Enable SCIM (Recommended for Enterprise)

For enterprise IDPs like Okta and Azure AD, enable SCIM for real-time user provisioning and immediate deprovisioning. SCIM is available on all plans.

1. In **Developer Portal** → the app's **Identity** page → **SCIM** section
2. Click **Enable SCIM**
3. **Copy the bearer token** (shown only once!) and the SCIM endpoint URL
4. Configure the IDP to push to this endpoint (see [SCIM Reference](/identity-providers/scim) for per-IDP instructions)

<Warning>
**Save the bearer token immediately!** The SCIM bearer token is displayed only once when you enable SCIM. If you lose it, you can rotate it from the Developer Portal, but the old token will be invalidated.
</Warning>

### Test Deprovisioning

To verify the integration works end-to-end:

1. Assign a test user to the application in the IDP
2. Verify the user appears in Alter Vault (check the SCIM sync status in Developer Portal)
3. Remove the test user from the IDP
4. Verify their OAuth grants are revoked in Alter Vault

</Steps>

## What Happens Behind the Scenes

When you pass a JWT or when your IDP sends a SCIM event:

1. **User identity is resolved** — Alter Vault maps the JWT `sub` claim (or SCIM `externalId`) to an internal user record
2. **Groups are synced** — Group memberships from JWT claims or SCIM group operations are synced to Alter Vault
3. **Connections are linked** — OAuth grants are associated with the user for lifecycle management
4. **Policies are evaluated** — Group-based access policies can restrict which providers a user can access

When a user is **deprovisioned**:

1. User status is set to `deprovisioned`
2. All linked OAuth grants are immediately revoked
3. All stored tokens are permanently deleted
4. Subsequent token requests for this user are denied
