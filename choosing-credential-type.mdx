## Quick Decision Guide

```
Do your END USERS need to authorize access to THEIR accounts?
├── Yes → Use OAuth Connections
│         (e.g., a user connects their Google Calendar)
└── No  → Use Managed Secrets
          (e.g., you store your Stripe API key)
```

<Info>
**Both use the same `vault.request()` method.** The only difference is where the `connection_id` comes from: OAuth returns it when the user completes authorization; Managed Secrets returns it when you store a credential in the Developer Portal.
</Info>

## Comparison

| | OAuth Connections | Managed Secrets |
|---|---|---|
| **Best for** | End users authorizing access to **their** accounts | **Your** API keys and service tokens |
| **Who provides the credential** | End user (via OAuth flow) | Developer (via Developer Portal) |
| **Credential is per...** | Per user | Per service |
| **`connection_id` comes from** | `onSuccess` callback or `connect()` return | Developer Portal |
| **Token refresh** | Automatic | Manual (re-store when rotated) |
| **User consent required** | Yes | No |
| **Visible in Wallet** | Yes (users can revoke) | No (backend-only) |
| **Setup complexity** | OAuth app + Alter Connect flow | Paste credentials in portal |

## Providers That Support Both

Some providers offer both OAuth (for end-user access) and API keys (for developer/server access). Here's when to use each, with our recommendation:

### GitHub

| Method | Use When | Recommendation |
|--------|----------|----------------|
| **OAuth** | Users authorize access to **their** repos, issues, and PRs | **Prefer for user-facing features** |
| **Managed Secret** (PAT) | Your backend needs server-level access to your org's repos | Use for CI/CD, automation, org-wide access |

<Tip>
**Recommendation: OAuth** for user-facing features. Use a managed secret (Personal Access Token) only when you need org-wide or service-level access that doesn't belong to a specific user.
</Tip>

### Stripe

| Method | Use When | Recommendation |
|--------|----------|----------------|
| **OAuth** (Stripe Connect) | Users connect **their** Stripe accounts to your platform (marketplace model) | Use for multi-merchant platforms |
| **Managed Secret** (API key) | Your backend processes payments using **your** Stripe account | **Prefer for most apps** |

<Tip>
**Recommendation: Managed Secret** for most applications. You typically process payments with your own Stripe account, not your users' accounts. Use OAuth (Stripe Connect) only if you're building a marketplace or platform where merchants connect their own Stripe accounts.
</Tip>

### Sentry

| Method | Use When | Recommendation |
|--------|----------|----------------|
| **OAuth** | Users connect **their** Sentry accounts for cross-org error monitoring | Use for multi-tenant monitoring platforms |
| **Managed Secret** (auth token) | Your backend monitors errors in **your** Sentry org | **Prefer for most apps** |

<Tip>
**Recommendation: Managed Secret** for most applications. You typically monitor your own Sentry organization. Use OAuth only if users need to connect their own Sentry orgs to your platform.
</Tip>

### Discord

| Method | Use When | Recommendation |
|--------|----------|----------------|
| **OAuth** | Users authorize your app to access **their** Discord identity, servers, or post on their behalf | **Prefer for user-facing features** |
| **Managed Secret** (bot token) | Your bot operates server-wide using a bot token | Use for Discord bots and server automation |

<Tip>
**Recommendation: OAuth** for user identity and user-scoped actions. Use a managed secret (bot token) for server-wide bot operations where no individual user context is needed.
</Tip>

### Airtable

| Method | Use When | Recommendation |
|--------|----------|----------------|
| **OAuth** | Users connect **their** Airtable bases to your app | **Prefer for user-facing features** |
| **Managed Secret** (PAT) | Your backend accesses **your** Airtable bases | Use for internal tools and automation |

<Tip>
**Recommendation: OAuth** when users bring their own Airtable data. Use a managed secret (Personal Access Token) for internal dashboards or automation that accesses your own bases.
</Tip>

### PagerDuty

| Method | Use When | Recommendation |
|--------|----------|----------------|
| **OAuth** | Users connect **their** PagerDuty accounts for incident management | Use for multi-tenant incident platforms |
| **Managed Secret** (API key) | Your backend manages incidents in **your** PagerDuty account | **Prefer for most apps** |

<Tip>
**Recommendation: Managed Secret** for most applications. You typically manage incidents in your own PagerDuty account. Use OAuth only if users need to connect their own PagerDuty accounts.
</Tip>

## Summary Table

| Provider | Prefer | Reason |
|----------|--------|--------|
| **GitHub** | OAuth | User-facing: users authorize their own repos |
| **Stripe** | Managed Secret | Most apps use their own Stripe account for payments |
| **Sentry** | Managed Secret | Most apps monitor their own Sentry org |
| **Discord** | OAuth (user) / Managed (bot) | Depends: user identity vs server-wide bot |
| **Airtable** | OAuth | User-facing: users bring their own bases |
| **PagerDuty** | Managed Secret | Most apps manage their own PagerDuty account |

## Still Unsure?

Ask yourself: **"Does my end user need to log in and grant permission?"**

- **Yes** → OAuth connection
- **No** → Managed secret

Both get the same security: encrypted storage, policy enforcement, audit logging, and zero token exposure in your code.
