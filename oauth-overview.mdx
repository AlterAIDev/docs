## What Are OAuth Connections?

OAuth grants let **end users** authorize access to their accounts on third-party services like Google, Slack, GitHub, and more. When a user completes the OAuth flow, Alter Vault stores their tokens securely and provides a `grant_id` to make API calls on their behalf.

Application code never sees or handles OAuth tokens. A call to `vault.request(method, url, grant_id=grant_id)` is all that is needed and the SDK injects the right credential automatically, refreshing expired tokens transparently.

## When to Use OAuth

Use OAuth grants when:

- **End users** need to authorize access to **their** accounts (e.g., a user's Google Calendar, a user's Slack workspace)
- You need **per-user** credentials (each user gets their own `grant_id`)
- The service supports OAuth 2.0 and you want automatic token refresh
- Users should be able to see and revoke their connections via the [Wallet Dashboard](https://wallet.alterauth.com)

## How It Works

<Steps>
  <Step title="User Authorizes">
    The end user completes the OAuth flow via Alter Connect (popup, redirect, or headless). They log into their account and grant permissions.
  </Step>

  <Step title="You Get a grant_id">
    The `onSuccess` callback (or `connect()` return value) provides a `grant_id` (UUID) for the user's connected account. Store this in the application database mapped to the user.
  </Step>

  <Step title="Tokens Stored Securely">
    Access tokens, refresh tokens, and expiry data are encrypted (AES-256-GCM) and stored in Alter Vault -- never in the application database.
  </Step>

  <Step title="Make API Calls">
    Call `vault.request(method, url, grant_id=grant_id)` -- the SDK retrieves the token, refreshes it if needed, and injects it as the correct auth header.
  </Step>
</Steps>

## Quick Example

<CodeGroup>

```python Python
from alter_sdk import App, HttpMethod

async with App(
    api_key="alter_key_...",
    caller="my-agent",
) as vault:
    # grant_id from Alter Connect (stored in your DB)
    response = await vault.request(
        HttpMethod.GET,
        "https://www.googleapis.com/calendar/v3/calendars/primary/events",
        grant_id=grant_id,
        query_params={"maxResults": "10"},
    )
    events = response.json()
```

```typescript TypeScript
import { App, HttpMethod } from "@alter-ai/alter-sdk";

const vault = new App({
  apiKey: "alter_key_...",
  caller: "my-agent",
});

// grantId from Alter Connect (stored in your DB)
const response = await vault.request(
  HttpMethod.GET,
  "https://www.googleapis.com/calendar/v3/calendars/primary/events",
  { grantId, queryParams: { maxResults: "10" } },
);
const events = await response.json();
await vault.close();
```

</CodeGroup>

## Three Ways to Trigger OAuth

| Method | Best for | How it works |
|--------|----------|--------------|
| **Headless** (`vault.connect()`) | CLI tools, scripts, Jupyter notebooks, AI agents | One SDK call does everything — creates a session, opens the browser, and polls until complete. No frontend needed. |
| **Redirect** | Server-rendered apps, email links | Backend calls `create_connect_session()` and redirects the user to the returned `connect_url`. After OAuth, the user is sent back to the `return_url`. No JS SDK needed. |
| **Popup** (`@alter-ai/connect`) | SPAs (React, Vue, etc.) | Backend calls `create_connect_session()` and passes the `session_token` to the frontend SDK, which opens a popup with `onSuccess`/`onError` callbacks. |

See the [Quickstart](/quickstart#choosing-an-oauth-trigger-method) for full examples of each method.

## Key Features

- **Automatic token refresh** -- tokens are refreshed transparently before they expire
- **Grant policy (TTL)** -- set maximum grant lifetimes with `grant_policy`
- **Per-user grants** -- each user who authorizes gets their own `grant_id`
- **Multi-account support** -- users can connect multiple accounts to the same provider (e.g., personal + work Gmail)
- **Agent delegation** -- a user can authorize a named agent to call providers on their behalf without sharing the underlying OAuth tokens
- **Wallet visibility** -- users can see and revoke their connections (and agent delegations) via the Wallet Dashboard
- **[Policy enforcement](/reference/policies)** -- time-based and IP-based access rules on every token retrieval
- **[Full audit trail](/reference/audit-logs)** -- every token access is logged with actor identity, method, URL, and reason

## Agent Delegation

When a user consents at the Connect UI, the resulting OAuth grant can be authorized for use by a specific managed agent. The agent calls providers on the user's behalf via its own API key — the user's OAuth tokens never leave the backend, and the agent's access can be revoked independently of the underlying connection.

**Operator-initiated delegation** — an `App` creates a Connect session with `agent` set:

```python
session = await app.create_connect_session(
    allowed_providers=["slack"],
    agent="customer-success-bot",  # UUID or managed-agent name
)
```

The Connect UI banner names the agent on the consent screen. After the user authorizes, the agent can call Slack via `agent.request(provider="slack")` — no `grant_id` needed; the SDK resolves the user's grant through the delegation row.

**Agent-initiated delegation** — an `Agent` constructed with a `user_token_getter` calls `create_connect_session()` with no `agent` argument. The agent delegates to itself; the user consents to "give this specific agent access to my Slack."

**Disambiguation** — when an agent has delegations from multiple users on the same provider, the caller passes a disambiguator on the per-call `Agent.request`:

```python
# JWT-authenticated path
await agent.request(method, url, provider="slack", user_token=jwt)

# Asserted UUID path (skips JWT validation)
await agent.request(method, url, provider="slack", app_user_id=user_id)
```

The JWT is a *disambiguator only* — it picks among delegations the agent already holds. An agent presenting a JWT for a user it has no delegation for sees `GrantNotFoundError`, not a successful token.

**Ambiguity recovery** — calling without a disambiguator when multiple delegated users exist raises `AmbiguousGrantError`. The exception carries `app_user_ids` (UUIDs only — no emails or names, to avoid leaking user identity across tenants):

```python
from alter_sdk import AmbiguousGrantError

try:
    await agent.request(method, url, provider="slack")
except AmbiguousGrantError as e:
    # Pick one of the candidate users and retry. UUIDs only — the
    # SDK consumer is expected to know how to render them (e.g.
    # by joining against an app-side user table).
    chosen_user_id = pick_user_from(e.app_user_ids)
    await agent.request(method, url, provider="slack", app_user_id=chosen_user_id)
```

For the related "agent has no access path at all" case (no delegation AND no managed-secret grant for the provider), catch `NoDelegatedGrantError` instead — the recovery is to run a Connect session with `agent=` set so a user can consent.

**Revocation** — both classes expose `revoke_delegation`:

```python
await app.revoke_delegation(grant_id, agent_id)  # operator path
await agent.revoke_delegation(grant_id)          # agent self-revoke
```

The OAuth grant itself stays active — only the agent's access path is removed. Users can also revoke delegations from the Wallet Dashboard's "Delegated agents" section on each connection.
