*Code that runs as a managed agent uses the `Agent` class.* This page covers the runtime surface; for provisioning and key management, see [Creating agents](/agents/creating-agents) and [Keys and rotation](/agents/keys-and-rotation).

## Constructing an Agent

Two ways:

<CodeGroup>
```python Python
# (a) Direct: per-agent API key from env
from alter_sdk import Agent

agent = Agent(api_key=AGENT_API_KEY)

# (b) From an App (impersonation): app key + agent UUID
from alter_sdk import App

app = App(api_key=APP_API_KEY)
agent = app.get_agent(agent_id)
```

```typescript TypeScript
// (a) Direct
import { Agent } from "@alter-ai/alter-sdk";

const agent = new Agent({ apiKey: AGENT_API_KEY });

// (b) From an App
import { App } from "@alter-ai/alter-sdk";

const app = new App({ apiKey: APP_API_KEY });
const agent = app.getAgent(agentId);
```
</CodeGroup>

Both shapes produce the same backend identity. See [Agents overview](/agents/overview) for when to pick which.

## Identifying the agent at runtime

<CodeGroup>
```python Python
me = await agent.me()
print(me.id, me.name, me.scopes, me.policy)
```

```typescript TypeScript
const me = await agent.me();
console.log(me.id, me.name, me.scopes, me.policy);
```
</CodeGroup>

`me()` returns the agent's `AgentInfo`. Useful for self-introspection in tools that need to display "who am I" without a separate config plumbing step. Only available on `Agent`, not `App`.

## Tracing a run

`trace()` binds a `run_id` to every request inside its scope. The run ID flows into the audit log so a multi-call workflow can be reconstructed:

<CodeGroup>
```python Python
async with agent.trace(run_id="run-abc123"):
    await agent.request("GET", url1, grant_id=g1)
    await agent.request("POST", url2, grant_id=g2)
    # both audit rows tagged with run_id="run-abc123"
```

```typescript TypeScript
await agent.trace({ runId: "run-abc123" }, async () => {
  await agent.request("GET", url1, { grantId: g1 });
  await agent.request("POST", url2, { grantId: g2 });
});
```
</CodeGroup>

Pair with `context={...}` for per-call metadata — `trace()` for the run-level ID, `context` for tool name / step name.

To attach run/tool/thread metadata without using `trace()`, pass `context={"run_id": ..., "tool": ...}` directly on `agent.request(...)`.

## Bootstrapping a grant from the agent

Agents can run `agent.connect()` and `agent.authenticate()` the same way an `App` can — useful for self-service flows where the agent obtains its own delegation rather than waiting for an operator to bind one:

<CodeGroup>
```python Python
results = await agent.connect(providers=["github"])
print(results[0].grant_id)
```

```typescript TypeScript
const results = await agent.connect({ providers: ["github"] });
console.log(results[0].grantId);
```
</CodeGroup>

The resulting grant is bound to the agent (not to a user). See [Headless OAuth](/oauth/headless).

## Self-revoking a delegation

When an agent decides it should no longer hold a delegation:

<CodeGroup>
```python Python
await agent.revoke_delegation(grant_id)
```

```typescript TypeScript
await agent.revokeDelegation(grantId);
```
</CodeGroup>

Removes only the agent's access path; the underlying user grant is untouched. To revoke the underlying grant entirely, an operator must call `App.revoke_grant()` — see [Managing grants](/oauth/managing-grants).

## Picking the right class

For agent code that runs inside a user session — handling a request on behalf of a specific logged-in user — construct `App(caller=AGENT_UUID, user_token_getter=...)` rather than `Agent`. The `App` shape attributes the call to the user; `Agent` attributes it to the workload identity. See [Agents overview](/agents/overview).
