## How Alter Vault Works

Alter Vault is a secure credential management service that handles OAuth tokens and API secrets so you don't have to. We store, refresh, and inject credentials automatically, while you focus on building your product.

### Two Paths, One API

Alter Vault supports two types of credentials, both accessed through the same `vault.request()` method:

- **OAuth Connections** — for third-party services where users authorize access (Google, Slack, GitHub)
- **Managed Secrets** — for your own APIs where you already have the credentials (API keys, service tokens)

```mermaid
graph LR
    A1[User Authorizes] --> B1[OAuth Flow]
    B1 --> C[Credentials Encrypted in Vault]
    A2[Developer Stores Secret] --> C
    C --> D[vault.request]
    D --> E[Credential Injected Automatically]
    E --> F[API Response Returned]
```

## Zero Credential Exposure

Your application **never sees or stores credentials**. Here's what happens instead:

<Steps>
  <Step title="Connect or Store">
    **OAuth**: User authorizes through Alter Connect UI. **Managed Secrets**: Developer stores credentials via the Developer Portal.
  </Step>

  <Step title="We Encrypt & Store">
    All credentials are encrypted (AES-256-GCM) and stored in our secure vault
  </Step>

  <Step title="You Call vault.request()">
    The SDK retrieves and injects credentials into API calls automatically
  </Step>

  <Step title="Automatic Management">
    OAuth tokens refresh before expiring. Managed secrets are injected with the correct header format.
  </Step>
</Steps>

## Security

### Encryption
- **AES-256-GCM** encryption for all stored tokens
- **TLS 1.3** for all data in transit
- **Hardware security modules** for key management

### Access Control
- **API key authentication** for your backend
- **Session tokens** for frontend (never expose API keys)
- **Policy enforcement** (time-based access, IP allowlist; scopes enforced via provider configuration)
- **Rate limiting** to prevent abuse
- **Fail-closed design** — if the policy service is unavailable, all token access is denied

### Compliance
- **SOC 2 Type II** certified
- **HIPAA** compliant infrastructure
- **GDPR** compliant with data residency options
- **PCI DSS** Level 1 for payment integrations

## Connection ID: The Universal Handle

Every credential in Alter Vault has a unique `connection_id` (UUID). Once you have one, usage is identical via `vault.request()`. The key difference is **where it comes from**:

| Credential Type | Where `connection_id` Comes From | Who Creates It |
|---|---|---|
| **OAuth** | `onSuccess` callback when end user completes OAuth via Alter Connect | **End user** (by logging into Google, Slack, etc.) |
| **Managed Secret** | Developer Portal when you store a credential | **Developer** (via portal UI) |

```python
# OAuth connection — connection_id from end user completing Alter Connect
# (stored in your DB, mapped to your user)
oauth_conn_id = db.get_connection(user_id="alice", provider="google")
response = await vault.request(
    oauth_conn_id,
    HttpMethod.GET,
    "https://www.googleapis.com/calendar/v3/calendars/primary/events",
)

# Managed secret — connection_id from Developer Portal
# (stored in your config/env vars, same for all users)
response = await vault.request(
    os.environ["LOYALTY_API_CONNECTION_ID"],
    HttpMethod.GET,
    "https://api.internal.com/v1/loyalty/points",
)
```

<Note>
**OAuth connections are per-user** — each end user who authorizes gets their own `connection_id`. **Managed secrets are per-service** — you store one credential and all your backend code shares the same `connection_id`.
</Note>

### Connection Deduplication

Alter Vault automatically prevents duplicate OAuth connections for the same account. When a user re-authorizes, we **update** the existing connection instead of creating a duplicate.

```mermaid
graph TD
    A[User Re-authorizes] --> B{Check account_identifier}
    B -->|Found| C[Update Tokens on Existing Connection]
    B -->|Not Found| D[Create New Connection]

    C --> E[Return connection_id]
    D --> E
```

- Search by the OAuth account identifier (email, sub, or ID from the provider)
- If found, update the existing connection with fresh tokens
- If not found, create a new connection and return the new `connection_id`

<Note>
If a user re-authorizes with the same OAuth account, the existing connection is updated. The `connection_id` remains the same.
</Note>

## Credential Management

We handle the entire credential lifecycle:

### What We Do Automatically

- ✅ **Store credentials securely** — Encrypted at rest (AES-256-GCM)
- ✅ **Refresh OAuth tokens before expiry** — No failed API calls
- ✅ **Inject the right header** — Bearer, API Key, Basic Auth — automatic
- ✅ **Enforce policies** — Time-based and IP restrictions on every access
- ✅ **Retry on failure** — Automatic error recovery
- ✅ **Audit all access** — Complete compliance trail for both OAuth and managed secrets

### What You Never Do

- ❌ Store tokens or API keys in your database
- ❌ Implement refresh logic
- ❌ Construct auth headers manually
- ❌ Manage encryption keys
- ❌ Build retry mechanisms

## Integration Example

Here's the complete flow in practice:

### 1. Frontend: User Connects (OAuth)

```javascript
// User clicks "Connect Google" — your backend creates a session (no user_id needed)
const { session_token } = await fetch('/api/session').then(r => r.json());
await alterConnect.open({
  token: session_token,
  onSuccess: (connections) => {
    // User completed OAuth! Each connection has a connection_id.
    // Store these in your database, mapped to your user.
    for (const conn of connections) {
      saveToDb(currentUser.id, conn.connection_id, conn.provider);
    }
  }
});
```

### 2. Backend: Use the Connection

```python
from alter_sdk import AlterVault, ActorType, HttpMethod

vault = AlterVault(
    api_key="alter_key_...",
    actor_type=ActorType.BACKEND_SERVICE,
    actor_identifier="my-service",
)

# OAuth: connection_id from your DB (stored when user completed Alter Connect)
connection_id = db.get_connection(user_id="alice", provider="google")
response = await vault.request(
    connection_id,
    HttpMethod.GET,
    "https://www.googleapis.com/calendar/v3/calendars/primary/events",
)
events = response.json()

# Managed secret: connection_id from Developer Portal (stored in your config)
response = await vault.request(
    os.environ["LOYALTY_API_CONNECTION_ID"],
    HttpMethod.GET,
    "https://api.internal.com/v1/loyalty/points",
)
```

That's it. No token handling, no refresh logic, no storage concerns.
