## Getting Started

The Developer Portal ([portal.alter.com](https://portal.alter.com)) is where you configure OAuth providers, set security policies, and manage your application.

<Steps>
  <Step title="Create Account">
    Sign up at [portal.alter.com](https://portal.alter.com)
  </Step>

  <Step title="Create Application">
    Each app has its own OAuth connections and API keys
  </Step>

  <Step title="Add OAuth Providers">
    Configure Google, Slack, Microsoft, etc.
  </Step>

  <Step title="Copy Your API Key">
    Get your API Key (`alter_key_...`) for the SDK
  </Step>
</Steps>

## OAuth Provider Setup

### Add Your OAuth Apps

For each provider you want to support:

1. Go to **OAuth Providers** → **Add Provider**
2. Enter credentials from the provider's console:
   - **Client ID**
   - **Client Secret**
   - **Redirect URI**: Always use `https://api.alter.com/oauth/callback`
3. Select the scopes you need
4. Save and activate

### Provider Consoles

| Provider | Where to Get Credentials |
|----------|-------------------------|
| Google | [console.cloud.google.com](https://console.cloud.google.com) |
| Microsoft | [portal.azure.com](https://portal.azure.com) |
| Slack | [api.slack.com/apps](https://api.slack.com/apps) |
| GitHub | [github.com/settings/apps](https://github.com/settings/apps) |

## Attribution

Define how you identify users for OAuth connections:

1. Go to **Application** → **Attribution**
2. Add your user identifier fields:
   ```json
   // Single user
   {"user_id": "string"}

   // Multi-tenant
   {"org_id": "string", "user_id": "string"}

   // Team shared
   {"team_id": "string"}
   ```
3. Mark required fields
4. Save

This must match what you pass in your SDK calls.

## Custom Schemas

Add structured data to any OAuth connection for enhanced functionality.

### Use Cases
- **Sync Settings**: Which data to sync, how often
- **User Preferences**: Per-connection configuration
- **Metadata**: Labels, tags, categories
- **Internal Mapping**: Link to your database records

### Creating a Schema

1. Go to **Application** → **Custom Schemas**
2. Click **Create Schema**
3. Use the visual builder to add fields:
   ```json
   {
     "sync_preferences": {
       "calendar": {
         "enabled": true,
         "frequency": "hourly"
       },
       "email": {
         "enabled": false
       }
     },
     "department": "engineering",
     "cost_center": "CC-001"
   }
   ```
4. Set field types and validation:
   - **Types**: string, number, boolean, object, array
   - **Validation**: required, min/max, enum values
   - **Defaults**: Set default values
5. Save and apply to connections

### Using Schema Data

```python
# Custom schema data is set during the OAuth connection flow
# via the Alter Connect session endpoint, then available
# in the Alter Vault dashboard for filtering and management.
#
# The SDK's vault.request() uses attribution (user dict) to
# find the right connection. Custom schema data enriches
# the connection metadata for your dashboard workflows.
```

## Security Policies

Control how and when OAuth tokens can be accessed. Policies are enforced in real-time by [Cerbos](https://cerbos.dev) before tokens are returned from the API.

<Note>
  Policies can only be configured for OAuth providers that you've already set up in the **OAuth Providers** tab. Add a provider there first, then configure its policy.
</Note>

### Policy Rule Types

| Rule | Description | Example |
|------|-------------|---------|
| **Scope Restrictions** | Limit which OAuth scopes can be retrieved | Only allow `calendar.readonly` |
| **Time-Based Access** | Restrict to business hours or weekdays | Mon-Fri, 9am-5pm |
| **IP Allowlist** | Only allow access from specific IPs | Office network `203.0.113.0/24` |
| **Required Attributes** | Require connection metadata | Must have `user_id` and `org_id` |

### Creating Policies

<Steps>
  <Step title="Configure a Provider">
    Go to **App Details** → **OAuth Providers** and add at least one provider (e.g., Google)
  </Step>

  <Step title="Open Policies Tab">
    Go to **App Details** → **Policies**. Only your configured providers appear here.
  </Step>

  <Step title="Add Policy">
    Click **Add Policy** next to a provider and configure rules:

    - **OAuth Scopes**: Check which scopes are allowed (if none checked, all scopes are permitted)
    - **Time Restrictions**: Check "Business hours only" (9am-5pm) or "Weekdays only" (Mon-Fri)
    - **IP Allowlist**: Add IPv4/IPv6 addresses or CIDR ranges
    - **Required Attributes**: Add attribute keys that must exist on the connection
  </Step>

  <Step title="Save">
    Click **Save Policy**. Changes take effect immediately for new token retrievals.
  </Step>
</Steps>

### Policy Examples

**Scope restriction** — only allow read-only calendar access:
```json
{
  "allowed_scopes": ["openid", "profile", "https://www.googleapis.com/auth/calendar.readonly"]
}
```

**Business hours + IP allowlist** — restrict to office network during work hours:
```json
{
  "time_restrictions": { "business_hours_only": true, "weekdays_only": true },
  "ip_allowlist": ["203.0.113.0/24"]
}
```

**Required attributes** — ensure every token access is attributed to a user and org:
```json
{
  "required_attributes": ["user_id", "org_id"]
}
```

### Policy Evaluation

All configured rules are evaluated with **AND logic** — every rule must pass for token access to be granted.

1. Scope restrictions (are the connection's scopes in the allowed list?)
2. Time restrictions (is it within business hours / a weekday?)
3. IP allowlist (is the client IP in the list?)
4. Required attributes (does the connection have all required attributes?)

If **any** rule fails, access is denied with a specific reason. If **no policy** is configured for a provider, all token accesses are allowed (permissive default).

### Fail-Closed Security

If the policy service (Cerbos) is unavailable, the system **denies all token access** rather than bypassing policies. This ensures security even during outages.

## API Keys

### Get Your API Key

1. Go to **Application** → **API Keys**
2. Copy your **API Key**: `alter_key_...`

### Security Best Practices

- **Backend Only**: Never put API keys in frontend code
- **Session Tokens**: Create temporary tokens for frontend
- **Rotate Keys**: Generate new keys every 90 days
- **IP Restrictions**: Add allowlists for production
- **Environment Variables**: Never commit keys to code

## Connection Management

View and manage all OAuth connections:

### Connections Dashboard

- **Filter**: By provider, user, date, or custom schema fields
- **Search**: Find specific users or attributes
- **Bulk Actions**: Update or revoke multiple connections
- **Export**: Download connection data as CSV

### Connection Details

For each connection, view:
- Provider and account info
- Attribution data
- Custom schema data
- Token status and refresh history
- Audit logs of all access

### Revoke Access

1. Find the connection
2. Click **Revoke**
3. Choose:
   - **Soft Revoke**: Mark inactive (can reactivate)
   - **Hard Revoke**: Delete completely

## Organizations & Teams

### Team Management

1. Go to **Organization** → **Team**
2. Invite members with roles:
   - **Admin**: Full access
   - **Developer**: Manage apps and connections
   - **Viewer**: Read-only access

|------------|---------|--------------|

- **Filter** by event type, provider, user, actor, or date range
- **Search** for specific connections, users, or AI agents
- **Click** any entry for full details
- **Export** as CSV for compliance reporting

## Quick Reference

### Backend Code
```python
from alter_sdk import AlterVault, Provider, HttpMethod

# Initialize SDK with your API key from portal
vault = AlterVault(api_key="alter_key_...")

# Make API calls - token injected automatically
response = await vault.request(
    Provider.GOOGLE,
    HttpMethod.GET,
    "https://www.googleapis.com/calendar/v3/calendars/primary/events",
    user={"user_id": "alice"},  # Must match attribution config
)
events = response.json()
```

### Frontend Code
```javascript
// Create session on YOUR backend (using API key)
const { session_token } = await fetch('/api/oauth/session');

// Open OAuth flow (no API key needed)
await alterConnect.open({
    token: session_token,
    onSuccess: (connection) => {
        console.log('Connected!', connection);
    }
});
```
