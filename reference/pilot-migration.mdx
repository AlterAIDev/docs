## Overview

Before scoped keys shipped, Alter App API keys were effectively master keys ŌĆö a single secret that, once authenticated, could call every SDK method. Scoped keys introduced explicit per-key **scopes** that gate which SDK methods each key can call.

<Info>
**Existing keys keep working.** The upgrade is transparent ŌĆö the legacy `alter_key_*` plaintext continues to authenticate with full access (`["*"]`) so every existing call works. There is no forced rotation deadline; migration is at-leisure.
</Info>

## What changed

| Before | After |
|---|---|
| Single `alter_key_*` plaintext format | Two formats coexist ŌĆö legacy `alter_key_*` and scoped `alter_<env>_<type>_*` |
| No per-key scope storage | Each key carries an explicit scope set |
| Every call reached every SDK method | Every SDK method is gated by a scope check |
| No per-request attenuation | `with_constraints` / `withConstraints` macaroon-style narrowing |
| Single env (live) | `live` / `test` env tagged on every key |

The SDK transparently handles both key formats ŌĆö `App(api_key="alter_key_...")` and `App(api_key="alter_live_rk_...")` both work.

## When to rotate

Existing keys continue to work as `*`-scope keys. Rotate when:

- **Least-privilege is desired.** A key with `["*"]` can reach every SDK method; a scoped key can only do what it's explicitly granted. Tightening the scope set narrows the blast radius of a leak.
- **Per-request attenuation is needed.** `with_constraints` only narrows from the key's stored set ŌĆö a `*` key can be narrowed to anything; a tight key already has the minimum.
- **CIDR allowlisting at the key level is required.** Scoped keys carry their own `cidr_allowlist`; legacy keys rely on org-level network controls.
- **Calls start raising `InsufficientScopeError` with `scope_version_mismatch=True`.** A new scope was added at a newer catalog version; the key's pinned version doesn't include it. Rotate to pick up the new scope grammar.

## Migration steps

<Steps>
  <Step title="Identify the minimum scope set">
    Pick the smallest set of scopes that covers the calls this key actually makes. See the [SDK method scope requirements](/reference/endpoint-scopes) for the per-method requirements.

    For an unknown workload, start with the [Developer Portal](https://portal.alterauth.com)'s "Recommended minimal scopes" panel ŌĆö it computes the actual exercised scope set from the last 30 days of audit data.
  </Step>

  <Step title="Mint a new scoped key">
    In the Developer Portal: **App ŌåÆ API Keys ŌåÆ New Key**. Select the scopes from step 1, set an expiration (default 90d, max per org policy), optionally add a CIDR allowlist, and copy the plaintext **once**.

    Or programmatically with an existing key that holds `keys:admin`:

    ```python
    new_key = await alter_app.keys.derive(
        scopes=["grants:read", "tokens:retrieve"],
        expires_in=86400 * 30,  # 30 days
    )
    print(new_key.api_key)  # store this; it won't be shown again
    ```
  </Step>

  <Step title="Roll out the new key">
    Deploy the new key alongside the old one. Both authenticate; the new one is scoped, the old one is still `*`.
  </Step>

  <Step title="Cut traffic over">
    Switch the application config to read the new key. Verify in the Portal's per-key usage analytics that calls are landing on the new key (last_used_at updates).
  </Step>

  <Step title="Revoke the legacy key">
    Once the new key has been live for at least one full deploy cycle and all expected call patterns are confirmed, revoke the legacy key via the Portal **or** programmatically:

    ```python
    await alter_app.keys.revoke(key_id=legacy_key_id)
    ```

    Revocation is instant and cascades to any derived (`dk`) keys minted from it.
  </Step>
</Steps>

## Backward compatibility

The SDK validator accepts **both** key shapes:

- Legacy: `alter_key_<random>` (pre-scoping).
- Scoped: `alter_<env>_<type>_<random>_<checksum>` where:
  - `env` Ōłł `live` / `test`
  - `type` Ōłł `rk` (restricted), `ak` (agent), `dk` (derived), `pk` (publishable, future).

Both shapes can coexist in the same codebase ŌĆö the SDK and backend handle the routing transparently.

<Warning>
**Don't put `alter_key_*` keys in production for new deployments.** They work, but they're `["*"]`-scoped ŌĆö every leak is a full credential-broker compromise. New code should mint scoped keys from the start.
</Warning>

## Rotation overlap

`vault.keys.rotate(...)` mints a same-scope successor and marks the old key deprecated. Both remain valid for a configurable overlap window (default 7 days, max 30) so the new secret can be rolled out without a cutover gap.

```python
result = await alter_app.keys.rotate(
    key_id=current_key.id,
    overlap_days=7,
)
new_plaintext = result.api_key  # store; old key still valid for 7d
```

After the overlap, the old key returns 401 ŌĆö schedule a deploy that switches the new plaintext into the runtime config before the overlap expires. If more time is needed, pass a larger `overlap_days` at rotation time.

## Common gotchas

**"Why is my key getting `InsufficientScopeError` on a call that used to work?"**

The method now requires a specific scope. The exception fields show what's missing:

```python
try:
    await alter_app.revoke_grant(grant_id=...)
except InsufficientScopeError as exc:
    print(exc.required)  # ["grants:admin"]
    print(exc.granted)   # ["grants:read"]
    print(exc.missing)   # ["grants:admin"]
```

Either grant the missing scope at the next rotation, or have the caller use a key that already holds it.

**"My `with_constraints` call is being rejected as a broadening attempt."**

`with_constraints(scopes=[...])` can only narrow the key's stored set, never broaden. If the constraint includes a scope the key doesn't already hold, the SDK rejects it. Remove the broader scope or mint a key that holds it.

**"The key authenticates but every call gets `InsufficientScopeError` with `scope_version_mismatch=True`."**

A new scope was added to the catalog after the key was minted. The key is pinned to its mint-time catalog version. Rotate the key to pick up the new grammar ŌĆö `vault.keys.rotate(key_id=...)` mints a successor at the current catalog version.
